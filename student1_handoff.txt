4. LLM RESPONDER (backend/agent/responder.py)
   - Sends alert context + system prompt to LLM
   - System prompt instructs LLM to choose one of three actions:
       * reorder        — stock critically low
       * notify_manager — price anomaly or high consumption
       * escalate       — critical situation needing human review
   - Parses JSON response and routes to correct action handler
   - On API auth failure (403/401): leaves alert as "new" so watcher retries
   - On other failures: marks alert acknowledged by "system"
   STATUS: DONE ✅ (LLM integration pending valid API key — see below)

--------------------------------------------------------
ALERT LIFECYCLE (as implemented)
--------------------------------------------------------

1. Station agent POSTs fuel data to /ingest every 5 seconds
2. Alert rules run against each new record
3. Fingerprint generated: MD5(station_id + fuel_type + alert_type)
4. If unresolved alert with same fingerprint exists → SKIP (deduplication)
5. Otherwise new alert saved with status="new"
6. Watcher polls every 30s for status="new" + severity="critical"
7. Responder sends alert context to LLM with system prompt
8. LLM returns JSON action: reorder / notify_manager / escalate
9. Action handler executes and marks alert as acknowledged/resolved
10. Operators can also manually update status via PATCH /alerts/{id}
